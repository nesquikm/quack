## Requirement

`scripts/smoke-test.sh` brings up the real Compose stack and *calls* every MCP
tool plus a hook, but its read-path coverage is shallow:

- the three read tools (`get_neighbors`, `path_between`, `recent_decisions`) are
  exercised only against **missing/empty** data;
- admin tools are only checked for "not an error";
- hooks are verified via the `accepted_total` **counter**, not by confirming the
  hooked content actually became retrievable.

The one genuine positive round-trip is `add_memory â†’ (real model) â†’ search_memory`.

This FR adds comprehensive full-stack **positive** round-trips proving each MCP
tool and each hook actually **stores and retrieves correct data end-to-end** â€”
`HTTP â†’ MCP â†’ Neo4j` and `hook â†’ /ingest â†’ extractor â†’ Neo4j â†’ read it back` â€”
**driven through the real cheap model** (the approved `/brainstorm` decision for
maximal realism).

**Hybrid structure (approved):** the bash `scripts/smoke-test.sh` keeps the
Compose lifecycle, the `MODEL` gate, and teardown; the comprehensive per-tool
assertions move into a new bun driver `scripts/smoke-assertions.ts` that the smoke
invokes. Because the extractor is non-deterministic, assertions are
**tolerant/semantic with retry/poll**, and the read tools use a
**discover-then-traverse** chain: seed via `add_memory`, poll `search_memory` to
find a real node id, then feed that id to `get_neighbors` / `path_between`.

## Technical Design

### Modules

- **`scripts/smoke-test.sh`** â€” replace the shallow read-tool checks (current `get_neighbors {missing}` / `path_between {a,b}` / `recent_decisions` shape checks). After minting `MTOK` (and when `MODEL=1`), call `bun "$ROOT/scripts/smoke-assertions.ts" "$URL" "$ADMIN" "$MTOK" "$SLUG"` and propagate its exit code. Keep the teardown trap, the `MODEL` gate, and the skip-when-docker-unreachable behavior. **Also derives the member token's row id** for the `revoke_token` round-trip â€” `docker compose exec -T quack bun -e '<select id from /data/auth.sqlite>'` â€” and exports it as `QUACK_SMOKE_MEMBER_TOKEN_ID` so the driver can positively exercise `revoke_token` (see below). The `id` query is best-effort; if it returns empty the driver skips `revoke_token` with a logged note (graceful when run standalone, off-Docker).
- **`scripts/smoke-assertions.ts`** (new bun driver) â€” a `/mcp` call helper (bearer auth + `accept: application/json, text/event-stream`, parse `result.content[0].text`), a poll/retry helper, and tolerant matchers. Orchestrates: the discoverâ†’traverse read round-trip (AC.2); the hook round-trips (fire the real hook entry under `plugins/quack/hooks/_lib/entry/` or POST `/ingest` with known envelopes) plus the meta-tool denoise negative check (AC.3); the admin data-effect assertions with the destructive lifecycle ordered last (AC.4). Emits `ok`/`no` lines; exits non-zero on any failure. Uses bun's built-in `fetch` â€” no new dependency.

  **Testable seam (`bun test` surface).** Because the real round-trips need a live stack + model, the driver is split so its *decision logic* is pure and unit-testable while the orchestration stays a thin I/O wrapper. Exported pure helpers (covered by `scripts/smoke-assertions.test.ts` against fixture JSON â€” no network): `parseMcpText(rpcJson)` (unwrap `result.content[0].text`, surface `isError`); `discoverNodeId(searchEnvelope)` (first `results[].id`); `hasNeighbor(neighborsEnvelope)` (â‰Ą1 `results`); `pathFound(pathEnvelope)` (no `no_path_found` warning + â‰Ą1 result); `decisionPresent(decisionsEnvelope, needle)`; `isGroundedAnswer(askEnvelope, needle)` (`mode_used: "planned"` + `<memory>` wrap + needle); `contentSurfaced(searchEnvelope, needle)`; `userListed` / `projectListed` / `countsReflect` / `tokenRejected` / `projectGone`; `buildHookEnvelope(kind, knownText)`; `isMetaToolEnvelope(env)`; and the run aggregator `summarize(results) â†’ exitCode` (any `no` â‡’ non-zero). The matchers are **tolerant** (case-insensitive substring, not exact equality) to absorb model non-determinism.

  **`revoke_token` token_id.** `revoke_token` takes a numeric `token_id` that **no MCP tool exposes** (the only source is the server's `auth.sqlite`). `smoke-test.sh` reads the member token's row id from the container DB and passes it via the `QUACK_SMOKE_MEMBER_TOKEN_ID` env var; the driver, when the var is a positive integer, runs the positive lifecycle (`revoke_token{token_id}` â†’ assert that token is subsequently rejected), and when it is absent/empty skips `revoke_token` with a logged note. The destructive lifecycle still runs **last** and `remove_member` remains the MCP-reachable revocation proof.

## Testing

This FR *is* test infrastructure, so its **full** verification is the smoke run
itself: with `QUACK_MODEL_*` configured and Docker up, every positive round-trip
passes; with `MODEL` unset (or Docker down) the suite prints `SKIP` and exits 0.

The `bun run test` surface (what the merge gate actually runs) is the driver's
**pure helper logic** â€” `scripts/smoke-assertions.test.ts` unit-tests the matchers,
`parseMcpText`, node-id discovery, the `no_path_found` / `no_full_text_match`
sentinels, the meta-tool/denoise predicate, hook-envelope construction, and the
`summarize` exit aggregator against fixture JSON (no network). The live
round-trips (AC.2â€“.5) are **not** `bun test`-able â€” they are proven by running
the smoke. The gate `bunx tsc --noEmit` must pass for the new TypeScript driver
(requires the `tsconfig.json` `include` addition above). Note `bun run test` does
**not** execute the smoke â€” it runs via the `smoke-test` skill or
`bash scripts/smoke-test.sh` directly.
