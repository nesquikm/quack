## Requirement

Ship a single self-contained binary `quack-hook` (built via `bun build --compile`) that runs as a Claude Code hook handler. It reads the hook payload from stdin (Claude Code's hook protocol), reads `QUACK_SERVER_URL` + `QUACK_TOKEN` (+ optional `QUACK_PROJECT_SLUG`) from environment, applies a **client-side redaction pass** (defense-in-depth — same default pattern set as FR-4NY6S1), POSTs the envelope to `/ingest` with a 1-second timeout, and exits 0 regardless of outcome (fire-and-forget — never break the Claude Code session).

A `quack-hook init <slug>` subcommand writes a per-project config skeleton and prints a Claude Code hooks-config snippet. v1 supports three hook kinds matching FR-4NY6S1 AC.1: `session_start`, `stop`, `post_tool_use`.

## Acceptance Criteria

- AC-S2D0Z5.1: Single binary `quack-hook` built from `src/hooks/quack-hook.ts` via `bun build src/hooks/quack-hook.ts --compile --outfile dist/quack-hook`. A new `package.json` script `"build:hook": "bun build src/hooks/quack-hook.ts --compile --outfile dist/quack-hook"` exists. The emitted binary is fully self-contained — no Bun runtime required at the install target.
- AC-S2D0Z5.2: Dispatch by first CLI arg: `quack-hook session_start` / `quack-hook stop` / `quack-hook post_tool_use`. An unknown kind ⇒ exit 0 silently (forward-compat — future Claude Code hook kinds we haven't wired here are no-ops, not failures).
- AC-S2D0Z5.3: Hook payload is read from stdin as JSON (Claude Code's protocol). If stdin is closed, empty, or fails JSON parse ⇒ exit 0 silently (one stderr log line at debug level).
- AC-S2D0Z5.4: Build the `HookEnvelope` shape from FR-4NY6S1 AC.1 (`{ kind, payload, project_slug?, ts }`) and POST to `${QUACK_SERVER_URL}/ingest` with `Authorization: Bearer ${QUACK_TOKEN}` + `Content-Type: application/json`. `QUACK_SERVER_URL` default `http://127.0.0.1:7474`. `ts` is the script's current `new Date().toISOString()`.
- AC-S2D0Z5.5: Fire-and-forget semantics: `AbortSignal.timeout(1000)` on the fetch; any error (network, timeout, non-2xx response) is logged to stderr `[quack-hook] error <kind>: <message>` AND the script **exits 0**. The Claude Code session is never broken by a hook server issue.
- AC-S2D0Z5.6: Client-side redaction pass: same default pattern set as FR-4NY6S1 AC.5 (OpenAI / GitHub / Slack tokens, Bearer, JWT, env-style assignments). Patterns are baked into the binary at build time AND can be extended via `QUACK_HOOK_REDACTION_PATTERNS` env var (comma-separated regex strings, appended to defaults). Defense-in-depth — the server-side pass still runs (FR-4NY6S1 AC.5), but client-side scrubbing reduces what crosses the wire.
- AC-S2D0Z5.7: Configuration resolution order: (1) explicit env vars (`QUACK_SERVER_URL`, `QUACK_TOKEN`, `QUACK_PROJECT_SLUG`); (2) per-project env file `~/.quack/projects/${QUACK_PROJECT_SLUG}.env` (parsed dotenv-style — `KEY=value`, comments via `#`); (3) if `QUACK_TOKEN` is still unresolved ⇒ exit 0 silently. The "silent disable" path is intentional — a missing token shouldn't pop errors in the developer's editor every time Claude Code does anything.
- AC-S2D0Z5.8: `quack-hook init <project-slug>` subcommand: (a) creates `~/.quack/projects/${slug}.env` with placeholder lines (`QUACK_TOKEN=…`, `QUACK_SERVER_URL=http://127.0.0.1:7474`, optional `QUACK_PROJECT_SLUG=${slug}`); refuses to overwrite an existing file with stderr `[quack-hook] init: ${slug}.env already exists` and exit 1; (b) prints to stdout a starter Claude Code hooks-config YAML snippet covering `session_start` / `stop` / `post_tool_use`, with the env file path filled in; (c) returns exit 0 on success.
- AC-S2D0Z5.9: `README.md` Deployment section is extended with a "Hooks installation" subsection covering: building the binary (`bun run build:hook`), copying to `~/.local/bin/quack-hook` (or any PATH location), running `quack-hook init <slug>`, and pasting the printed snippet into Claude Code's hooks config.
- AC-S2D0Z5.10: Performance budget: `quack-hook` completes in < 200 ms wall-clock on a developer laptop (NFR-1 fire-and-forget budget). Achievable because the fetch is async with a 1-second cap, the script body is small, and the compiled binary starts fast (Bun's `--compile` cold start is ~5-10 ms on Apple Silicon).
- AC-S2D0Z5.11: Tests in `src/hooks/quack-hook.test.ts` mock `fetch` and stdin:
  - Happy path for each of the three kinds: POST body matches `HookEnvelope`; Authorization header carries the token; URL is `${QUACK_SERVER_URL}/ingest`.
  - Missing `QUACK_TOKEN` ⇒ no fetch + exit 0.
  - stdin closed / empty / non-JSON ⇒ no fetch + exit 0.
  - Server 5xx ⇒ fetch made + stderr log + exit 0.
  - Timeout ⇒ fetch aborted at 1 s + stderr log + exit 0.
  - Redaction: stdin payload containing `sk-test1234567890abcdefg` ⇒ outbound body has `«REDACTED»` in that position.
  - Unknown CLI arg ⇒ no fetch + exit 0.
- AC-S2D0Z5.12: `tests/hook-binary.test.ts` integration test: compiles the binary via `bun build --compile`, spawns it against a stub `Bun.serve` instance, asserts (a) binary exits 0, (b) stub received the expected envelope shape, (c) `Authorization: Bearer ${TOKEN}` header present. Skipped automatically if the test runner can't execute the compiled binary (skip-on-platform-mismatch tag).
- AC-S2D0Z5.13: M2's `compose.yml` healthcheck and FR-BKPM28's Docker setup do NOT change — the hook is a client-side artifact, not part of the server stack. The binary lives in `dist/quack-hook` after `bun run build:hook` and is intentionally NOT included in the Docker image (build context excludes via `.dockerignore`).

## Technical Design

### Modules

- **`src/hooks/quack-hook.ts`** — entry point; parses argv; routes to `init` subcommand or hook-dispatch path.
- **`src/hooks/dispatch.ts`** — reads stdin, applies redaction, builds envelope, calls `postEnvelope`.
- **`src/hooks/post.ts`** — `postEnvelope(envelope, opts): Promise<void>` with `AbortSignal.timeout(1000)`; logs stderr on failure; always resolves (no throw).
- **`src/hooks/redact.ts`** — client-side redaction; re-exports the default pattern set from the shared module.
- **`src/shared/redaction_patterns.ts`** — single source of truth for the default redaction pattern list. Imported by both `src/extract/redact.ts` (server, FR-4NY6S1) and `src/hooks/redact.ts` (client). Refactor FR-4NY6S1's implementation to consume from here.
- **`src/hooks/config.ts`** — env-var + dotenv file resolution; returns `{ serverUrl, token, projectSlug? } | null`.
- **`src/hooks/init.ts`** — `quack-hook init <slug>` subcommand; writes the env-file placeholder; prints the YAML snippet.
- **`package.json`** — adds `"build:hook"` script.

### Dependencies added

None. `fetch`, `process.stdin`, file IO are all Bun built-ins. dotenv parsing is a 20-line implementation in `src/hooks/config.ts` (no `dotenv` package — keeps the binary small).

### Out of scope here

- `pre_tool_use` and other Claude Code hook kinds (forward-compat — script no-ops on unknown kinds).
- Server-side hook-kind validation (FR-4NY6S1 AC.1 lists the supported set in Zod).
- npm-publishable package distribution (the compiled binary is the v1 install artifact).
- Cross-platform Windows compile (focus on macOS + Linux for v1; Bun's `--compile` may emit Windows binaries but that's untested here).
- Automatic update of hook scripts when the server-side envelope shape changes (v1 binaries ship per-release; no auto-update mechanism).

## Testing

- `src/hooks/post.test.ts` — fetch mocked: success / 5xx / timeout / network error all resolve cleanly; stderr log on failures; happy path issues the correct Authorization header + body.
- `src/hooks/redact.test.ts` — each default pattern matches; custom patterns via env append correctly; deep-walk handles nested objects/arrays.
- `src/hooks/config.test.ts` — env vars win; dotenv file fallback; missing token returns null; comment + whitespace handling in dotenv.
- `src/hooks/dispatch.test.ts` — happy path per kind; stdin empty ⇒ no fetch; stdin malformed ⇒ no fetch; missing config ⇒ no fetch; redaction applied before post.
- `src/hooks/init.test.ts` — fresh init writes the env file + prints snippet; pre-existing file ⇒ exit 1 + no overwrite.
- `src/hooks/quack-hook.test.ts` — end-to-end CLI parsing (argv routing); init vs dispatch paths.
- `tests/hook-binary.test.ts` — compiled-binary integration against `Bun.serve` stub.

## Notes

- The "exit 0 silently when no token" behavior is intentional: developers cloning the repo without setting up hooks shouldn't see errors in their editor every time Claude Code does anything. The cost is: silent disable is undetectable from inside Claude Code. The README installation guide tells you to verify by tailing the server's request log after the first session.
- The 1-second timeout matches the NFR-1 fire-and-forget budget (< 200 ms target, 1 s hard cap). Local-network POSTs complete in < 50 ms typically; the cap only triggers when the server is down or pathologically slow.
- The compiled binary is single-file and ~30-50 MB (Bun runtime baked in). Acceptable for a developer-tool install; intentionally NOT included in the Docker image (`.dockerignore` excludes `dist/`).
- Shared redaction patterns between client and server (`src/shared/redaction_patterns.ts`) ensure parity. A pattern added server-side via `QUACK_REDACTION_PATTERNS` doesn't automatically propagate client-side; operators set `QUACK_HOOK_REDACTION_PATTERNS` separately. Documented in README.
- `quack-hook init` is intentionally minimal — it doesn't connect to the server to validate the token. Validation happens on the first real hook fire when the server returns 401, which the user sees once in stderr (then it goes silent). Connecting on init would tie the install flow to server availability.
- Future enhancement: support `pre_tool_use` for early-redaction signaling (hook could refuse to forward sensitive payloads). Out of scope for v1; trivially additive (add a CLI dispatch case + a `HookKind` literal-union member server-side).
- Bun `--compile` produces platform-specific binaries; the build script ships only the host-platform binary by default. Cross-platform CI matrices are out of scope; users on a different platform can `bun run build:hook` themselves with the source.
