## Requirement

A single admin-only MCP tool `server_status` returning a snapshot struct of server-wide health (uptime, version, queue stats, error counts by category, system counts). Same admin gate as FR-WSFVNP's management tools (`ADMIN_TOOLS` allowlist). Snapshot only — no time-series, no per-project visibility, no Prometheus endpoint. In M2 the queue counters return `null` (extractor not yet wired) and the active error categories are `auth_401` + `admin_403`; M3+ activates `extraction_failed` / `db_error` / `queue_full` as those modules land.

## Acceptance Criteria

- AC-956DT2.1: `server_status` is registered as an MCP tool on the `/mcp` endpoint and added to the `ADMIN_TOOLS` allowlist in `src/admin/index.ts`. Non-admin caller invoking it ⇒ HTTP 403 `{ error: "forbidden" }` via FR-WSFVNP's admin gate.
- AC-956DT2.2: The response carries a top-level `version: "v1"` literal string — distinct from the build-version field at `counts.server_version`. Breaking changes ship as a new tool `server_status_v2`, not by mutating the v1 shape.
- AC-956DT2.3: `uptime_seconds`: integer seconds since the server process started, computed as `Math.floor((Date.now() - startedAt) / 1000)` where `startedAt` is captured once at module-load time.
- AC-956DT2.4: `queue` field returns the struct `{ depth, oldest_pending_age_seconds, accepted_total, dropped_full_total }`. In M2 all four are `null` (extractor queue doesn't exist yet). M3 wires the real counters via `src/metrics/counters.ts`.
- AC-956DT2.5: `errors` returns `{ since_boot_total, by_category }`. `by_category` is `Record<string, number>` so adding new categories in M3 is non-breaking. M2 actively tracks `auth_401` (incremented from FR-HA2WTQ AuthMiddleware on every 401) and `admin_403` (incremented from FR-WSFVNP dispatch on every 403). `since_boot_total` equals the sum of `by_category` values.
- AC-956DT2.6: `counts` returns `{ users, projects, tokens_active, server_version }`. Each integer count is a `SELECT COUNT(*)` against `auth.sqlite` at request time (`tokens_active` filters `revoked_at IS NULL`). `server_version` reads `package.json` `version` once at startup (cached). `memories_total` is omitted in M2 (no graph DB yet); M3 adds the key.
- AC-956DT2.7: All in-process counters (`errors.since_boot_total`, `errors.by_category`, M3's queue totals) reset on server restart. No persistence — short-horizon snapshot, not a metrics store.
- AC-956DT2.8: `src/metrics/counters.ts` exports `incrementError(category: string): void`, `getSnapshot(): MetricsSnapshot`, and the `startedAt` constant. AuthMiddleware (FR-HA2WTQ §5) calls `incrementError('auth_401')` on every 401 return path; MCP dispatch (FR-WSFVNP §2) calls `incrementError('admin_403')` on every 403 return path. Reads are race-free on Bun's single-threaded runtime — no locks needed.
- AC-956DT2.9: Tool response time p95 < 50 ms on a 100-user / 10-project / 50-token seeded `auth.sqlite`. The two `COUNT(*)` queries dominate latency.

## Technical Design

### Modules

- **`src/metrics/counters.ts`** — tiny in-process counter store. `incrementError(category)` (Map<string, number>), `getSnapshot()` returns `{ errors: { since_boot_total, by_category }, queue: { … all null in M2 } }`, exports `startedAt = Date.now()` captured at module-load time.
- **`src/admin/tools/server_status.ts`** — MCP tool handler. Assembles response by reading counters, running the two `COUNT(*)` queries, and returning the assembled struct.

### Integration points (edits to existing FRs)

- FR-HA2WTQ §5 `src/auth/middleware.ts`: add `incrementError('auth_401')` on every 401 return path.
- FR-WSFVNP §2 `src/mcp/dispatch.ts`: add `incrementError('admin_403')` on every 403 return path.
- FR-WSFVNP §1 `src/admin/index.ts`: add `'server_status'` to the `ADMIN_TOOLS` set.
- FR-WSFVNP §3 tool-registration list: add `server_status` next to the other admin tools.

### Response shape (canonical, illustrative)

```ts
{
  version: "v1",
  uptime_seconds: 22132,
  queue: { depth: null, oldest_pending_age_seconds: null, accepted_total: null, dropped_full_total: null },
  errors: { since_boot_total: 12, by_category: { auth_401: 11, admin_403: 1 } },
  counts: { users: 4, projects: 6, tokens_active: 12, server_version: "0.0.1" }
}
```

### Dependencies added

None. Existing `bun:sqlite` + `Bun.file()` for package.json version read.

### Out of scope

- Time-series / metrics history (decided in `/brainstorm`).
- Per-project stats / member visibility (admin-only by design).
- `/metrics` Prometheus endpoint.
- `memories_total` count (depends on graph DB FR).
- Queue counters' real values (deferred to M3 when extractor exists).

## Testing

- `src/metrics/counters.test.ts` — increment / read; multiple categories; `since_boot_total` equals sum of `by_category` values; fresh instance starts at zero.
- `src/admin/tools/server_status.test.ts` — happy path returns full struct; admin-only via `ADMIN_TOOLS` gate; counts match seeded DB; M2 queue fields all `null`; unknown error category added to `by_category` appears in output without schema break; `version: "v1"` literal asserted.
- Integration test extending `src/server/index.test.ts`: trigger a 401 + a 403, then invoke `server_status` via the MCP HTTP client; assert `auth_401: 1, admin_403: 1, since_boot_total: 2`.
- `src/admin/tools/server_status.bench.test.ts` — p95 < 50 ms on seeded DB (AC-956DT2.9). Slow-runner guard tag.

## Notes

- The `version: "v1"` envelope is the lock-in escape hatch. If we need to break the shape later, ship `server_status_v2` as a separate tool; clients pick which they call. No coordinated upgrade required.
- Counter reset on restart is deliberate. The tool answers *"how is the server doing right now"*, not *"what's the all-time error count"*. Long-term metrics is a separate decision and probably a Prometheus endpoint, not this tool.
- Race-free counter reads rely on Bun's single-threaded runtime. If the extractor ever moves to a Worker (out of M2 scope), the counter store needs an `Atomics`-based variant — flagged as a future risk.
- The MCP tool description string should explicitly say "snapshot only, no streaming, no history" so Claude Code gives accurate answers when the operator asks something like "show me errors from the last hour" (the answer is "I can only show since-boot totals, sorry").
- The choice to read `package.json` version at startup (not per-request) avoids filesystem I/O on every snapshot call.
