## Requirement

Replace the direnv/`.envrc` per-workspace config mechanism with a single
project-scoped `.mcp.json` that `/quack:install` writes directly into the workspace
root. `.mcp.json` is the **one** config artifact: it declares the Quack MCP server
(literal token + server URL) and carries an `X-Quack-Sub-Project` header that is the
per-workspace sub-project tag — consumed server-side by `add_memory` (FR-A9BN0M AC.7),
and client-side by the hooks.

Three coupled changes:

1. **`/quack:install` writes `.mcp.json`** (not `.envrc`) and becomes **adaptive**: it
   always writes the config, and *additionally* mints the per-workspace token when an
   admin token is available — degrading to "operator supplies a token" when it is not.
   It derives a suggested sub-project from the git remote and interactively confirms it.
2. **The plugin's `mcp-servers/quack.json` is removed** — the server is declared only
   by the `.mcp.json` `/quack:install` writes, so there is exactly one declaration.
3. **The hooks read `.mcp.json`** (server URL + token + sub-project) instead of
   `process.env`, dropping the direnv dependency and the session-restart-for-env dance.

**MVP tradeoff (explicit, recorded, cheaply reversible):** `.mcp.json` carries the
*literal* per-workspace token and is **not** gitignored. The token is non-admin and
single-project — a leak grants read/write to one project's memory and nothing else.
The post-MVP path is a one-line change to the writer (emit `${QUACK_TOKEN}` + a
`.gitignore` line); no code reversal.

This FR is the client-side delivery half; the server-side `source` graph machinery is
FR-A9BN0M.

## Acceptance Criteria

- AC-55S220.1: `/quack:install` writes a project-scoped `.mcp.json` at the workspace
  root (resolved via `git rev-parse --show-toplevel`, falling back to `$PWD`). Shape:
  ```json
  { "mcpServers": { "quack": {
      "type": "http",
      "url": "<resolved server URL>/mcp",
      "headers": {
        "Authorization": "Bearer <literal per-workspace token>",
        "X-Quack-Sub-Project": "<confirmed sub-project slug>"
  } } } }
  ```
  When `.mcp.json` already exists, the `quack` entry is merged into the existing
  `mcpServers` object without disturbing sibling servers. When a `quack` entry already
  exists, `/quack:install` refuses to overwrite and prints the snippet for a manual
  merge (mirrors the current `.envrc` non-overwrite behaviour).

- AC-55S220.2: `/quack:install` is adaptive on token minting. When an admin token is
  available (`QUACK_ADMIN_TOKEN` in env, or supplied interactively), it mints the
  per-workspace `(user, project)` token via the existing idempotent admin-MCP flow
  (`create_project` → `register_user` → `add_member`). When no admin token is
  available, it prompts the operator to paste an already-issued per-workspace token.
  Either path yields the literal token written into `.mcp.json` per AC.1.

- AC-55S220.3: `/quack:install` derives a suggested sub-project slug:
  `git remote get-url origin` → URL basename → strip `.git` suffix → slug-normalize;
  when there is no remote, fall back to the workspace directory basename. It presents
  the suggestion interactively (`Sub-project [<derived>] — accept, or type an
  override`). A `--sub <name>` argument skips the prompt. The final value is validated
  against `^[a-z0-9][a-z0-9_-]{0,62}$`; a malformed override is refused with a message.
  The confirmed value is written into the `X-Quack-Sub-Project` header (AC.1).

- AC-55S220.4: `plugins/quack/mcp-servers/quack.json` is deleted, and any plugin
  manifest reference to it is removed. The Quack MCP server is declared **only** by the
  `.mcp.json` that `/quack:install` writes — a workspace that has not run
  `/quack:install` has no Quack MCP server (and no token, so the prior
  plugin-declared-but-401ing server bought nothing). Forward-only; no installed-user
  migration (`project_no_users_yet`).

- AC-55S220.5: The plugin hooks read configuration from `.mcp.json` instead of
  `process.env`. `plugins/quack/hooks/_lib/config.ts` is reworked to locate `.mcp.json`
  (walk up from `CLAUDE_PROJECT_DIR` / cwd to the workspace root), parse it, and extract
  from `mcpServers.quack`: the ingest base URL (the `url` value with a trailing `/mcp`
  stripped), the bearer token (`headers.Authorization` minus the `Bearer ` prefix), and
  the sub-project (`headers["X-Quack-Sub-Project"]`). The hook stamps the sub-project
  onto `HookEnvelope.sub_project` (FR-A9BN0M AC.1) and omits `project_slug` (the server
  resolves the project from the token — AC-4NY6S1.13). When `.mcp.json` is absent, has
  no `quack` entry, or is malformed JSON, the hook silently no-ops and exits 0 (the
  AC-44QGKH.2 silent-disable invariant).

- AC-55S220.6: The direnv / `.envrc` mechanism is removed from the install path.
  `/quack:install` no longer writes `.envrc`. `plugins/quack/README.md` and the
  repo-root `README.md` install flows drop the `.envrc` / `direnv allow` /
  `unset QUACK_ADMIN_TOKEN` steps and the `QUACK_SERVER_URL` / `QUACK_TOKEN` env-var
  table, and document the `.mcp.json` flow plus the single Claude Code session restart
  needed to pick up the new MCP server. `direnv` is removed from the prerequisites.

- AC-55S220.7: The committed-literal-token tradeoff is documented in three places:
  `/quack:install`'s closing output, the plugin `README.md`, and `technical-spec.md`
  (a new ADR / key-design-decision row). Each states that `.mcp.json` holds a literal
  non-admin single-project token, is committed by default for MVP, and names the
  post-MVP path (`${QUACK_TOKEN}` substitution + `.gitignore`).

- AC-55S220.8: Tests. `plugins/quack/commands/quack-install.md` shape is pinned by a
  test (the `.mcp.json` writer steps, the adaptive-mint branch, the sub-project
  derivation). The plugin shape / hermeticity tests (`tests/plugin-files.test.ts`,
  `tests/bundled-hooks-shape.test.ts`, `tests/plugin-install-local.test.ts`) are updated
  for the removed `mcp-servers/quack.json`. A new test covers the reworked `config.ts`
  `.mcp.json` reader (valid file, absent file, no `quack` entry, malformed JSON ⇒
  silent no-op) and the `.mcp.json` merge / non-overwrite behaviour.

- AC-55S220.9: `specs/requirements.md` traceability matrix is refreshed — rows added
  for every `AC-A9BN0M.*` and `AC-55S220.*` AC; the `AC-ZSN2GG.4` / `AC-ZSN2GG.5` rows
  that reference `mcp-servers/quack.json` and the `.envrc`-based `/quack:install` are
  updated to the new config-delivery shape. The cross-cutting `.envrc`/direnv prose in
  `requirements.md` § "Delivery model" and `technical-spec.md` §1 (Plugin packaging) is
  corrected to the `.mcp.json` mechanism.

## Technical Design

### /quack:install rewrite

`plugins/quack/commands/quack-install.md` is rewritten. New step order: parse slug +
`--sub` → resolve server URL → derive + confirm sub-project → resolve token (adaptive:
mint via admin MCP if an admin token is present, else prompt for a pasted token) →
write / merge `.mcp.json` → print follow-up (including the committed-token note). The
admin-MCP minting sub-flow is the existing idempotent
`create_project`/`register_user`/`add_member` sequence, now gated behind "admin token
available".

### .mcp.json as the single config artifact

Project-scoped `.mcp.json` is natively read by Claude Code. The same `mcpServers.quack`
object serves three consumers: Claude Code's MCP client (server connection), the hooks
(`config.ts` reads it for URL + token + sub-project), and — via the
`X-Quack-Sub-Project` header on every MCP request — the server's `add_memory`
(FR-A9BN0M AC.7). One file, three readers, no env vars, no direnv.

### Files

- Rewritten: `plugins/quack/commands/quack-install.md`.
- Deleted: `plugins/quack/mcp-servers/quack.json`.
- Reworked: `plugins/quack/hooks/_lib/config.ts` (env-reads → `.mcp.json` reads) +
  its `__tests__/config.test.ts`.
- Updated: `plugins/quack/README.md`, repo-root `README.md`, `technical-spec.md`,
  `requirements.md` (matrix + cross-cutting prose), and the plugin shape / hermeticity
  tests.

### Out of scope

- `${QUACK_TOKEN}` substitution + gitignoring `.mcp.json` — the post-MVP hardening;
  a one-line writer change, deferred deliberately.
- A separate `/quack:admin` minting command — the adaptive single command covers it.
- Multi-user-per-workspace token handling (a committed `.mcp.json` shares one token
  across cloners) — accepted MVP limitation, documented in Notes.

## Testing

Per AC.8. The `/quack:install` command is markdown (no unit-testable code) — its shape
is pinned by a presence/structure test, and the manual smoke (a live workspace + server)
is an M8 closeout step like AC-ZSN2GG.11. The `config.ts` reader is the substantive
unit-test target (valid / absent / no-`quack`-entry / malformed-JSON paths).

## Notes

- Removing the plugin's `mcp-servers/quack.json` is forward-only —
  `project_no_users_yet`, no migration shim (same posture as FR-44QGKH).
- The literal committed token is the deliberate MVP call from `/brainstorm`: a
  non-admin, single-project token in a (typically private, solo) repo. The reversal is
  cheap — emit `${QUACK_TOKEN}` and a `.gitignore` line — so no code is wasted if the
  project later wants the secret out of version control.
- A committed `.mcp.json` means every cloner of a shared repo gets the *same* token →
  shared identity for that project. Fine for the solo/MVP case; flagged so a future
  multi-dev FR can revisit.
- The hook now omits `project_slug` from the envelope entirely — the server resolves
  the project from the token (AC-4NY6S1.13), so the slug was always redundant when the
  token is correct. One fewer value to thread through `.mcp.json`.
